# 실패로 배우는 제약조건

성공만 해서는 구조를 완벽히 이해하기 어렵습니다. 실무 현장에서 **자주 발생하는 실패 사례 5가지**를 일부러 재현해 보며, TableDiff의 핵심 제약 조건을 몸으로 익혀봅시다.

---

## Case 1: 정렬 방향 불일치
**상황**: TableA는 `ORDER BY ID ASC`, TableB는 `ORDER BY ID DESC`로 설정.

* **결과**: 모든 데이터가 `OnlyInA`와 `OnlyInB`로 쏟아져 나옵니다.
* **이유**: 스트리밍 비교는 순서가 생명입니다. 순서가 다르면 같은 데이터(`ID=50`)가 있어도 서로 다른 위치에서 만나게 되어 "상대방에 없다"고 판단해 버립니다.
* **교훈**: **ORDER BY 절은 토씨 하나 틀리지 말고 똑같이 맞춰야 합니다.**

---

## Case 2: NULL 정렬 위치 차이
**상황**: 한쪽은 `NULLS FIRST`, 다른 쪽은 `NULLS LAST` (또는 생략 시 DBMS 기본값 차이).

* **결과**: NULL이 포함된 데이터부터 밀리기 시작해서 대량의 불일치가 발생합니다.
* **교훈**: DBMS마다 기본 정렬 방식이 다릅니다(Oracle은 NULL이 가장 큼, PG는 가장 큼 등). 반드시 명시적으로 `NULLS FIRST` 또는 `NULLS LAST`를 적어주는 습관을 들이세요.

---

## Case 3: SortKey 중복 (Unique 위배)
**상황**: `ORDER BY DEPT_NO` 처럼 중복이 많은 컬럼을 유일한 정렬키로 사용.

* **결과**: 1:1 매칭이 불가능해져서 비교 로직이 꼬이거나 에러가 발생할 수 있습니다.
* **교훈**: **SortKey는 반드시 유일(Unique)해야 합니다.** PK가 있다면 PK를 정렬키로 쓰는 것이 가장 안전합니다.

---

## Case 4: 타입 불일치 (문자 vs 숫자)
**상황**: TableA의 ID는 숫자 `100`, TableB의 ID는 문자열 `'100'`인 상태로 정렬.

* **결과**: `100`과 `10`의 정렬 순서가 다릅니다. (숫자: 10 < 100, 문자: '10' < '100' ... 하지만 '2'는 '10'보다 큼)
* **이유**: 정렬 순서가 뒤죽박죽이 되어 매칭에 실패합니다.
* **교훈**: 타입이 다르면 SQL에서 `CAST(COL AS NUMBER)` 등으로 **타입을 일치**시켜야 합니다.

---

## Case 5: 대용량 처리와 메모리
**상황**: LOB(대용량 텍스트/이미지) 컬럼을 비교하는데 `SELECT *`로 수만 건을 한 번에 조회.

* **결과**: `OutOfMemoryError` 발생.
* **교훈**: TableDiff는 스트리밍 방식이라 메모리 효율이 좋지만, LOB 데이터는 덩치가 큽니다.
    * 1. **JVM 힙 메모리 늘리기**: 실행 시 `-Xmx4g` 등으로 메모리를 확보하세요. (참고: [CLI 실행 방법](./cli.md))
    * 2. `batch` 사이즈를 줄이거나
    * 3. `WHERE` 절로 범위를 나누어(파티셔닝) 여러 번 실행하는 것이 현실적인 해결책입니다.
