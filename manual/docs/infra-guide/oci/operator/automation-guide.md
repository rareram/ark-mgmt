# OCI 운영 및 자동화 가이드

이 문서는 OCI 환경의 운영 효율성을 높이고, 반복적인 작업을 자동화하기 위한 핵심 전략과 모범 사례를 다룹니다.

## 1. IaC (Infrastructure as Code) 및 Landing Zone

### Landing Zone 도입

**Landing Zone**은 수작업으로 인한 오류를 줄이고, 보안 및 거버넌스 정책이 사전 적용된 표준화된 OCI 환경을 신속하게 배포하기 위한 핵심 전략입니다.

- **구성 요소**: 네트워크(VCN, Subnet), IAM 기본 정책, 태그 네임스페이스, 보안 그룹, 로깅 설정 등이 미리 코드로 정의된 템플릿입니다.
- **구현 도구**: **Terraform**을 사용하여 OCI 인프라를 코드로 관리하는 것을 권장합니다.
- **장점**:
    - **표준화**: 모든 팀이 동일한 기준의 인프라 환경에서 시작할 수 있습니다.
    - **신속성**: 새로운 프로젝트나 팀을 위한 환경을 몇 분 안에 배포할 수 있습니다.
    - **안전성**: 검증된 코드를 재사용하여 설정 오류의 위험을 최소화합니다.

### 리소스 생성 템플릿

팀의 자율성을 보장하면서도 거버넌스를 준수하게 하는 가장 좋은 방법은 **'리소스 생성 템플릿'**을 제공하는 것입니다.

- **예시**: 개발자가 VM 인스턴스를 생성할 때, 직접 모든 옵션을 설정하는 대신 'Dev-WebServer-Template'을 사용하도록 유도합니다.
- 이 템플릿에는 다음과 같은 항목이 사전에 정의되어 있습니다.
    - **컴파트먼트**: `cmp-dev`
    - **네트워크**: `vcn-dev`의 `private-subnet`
    - **필수 태그**: `Owner`, `Project` 등
    - **보안 옵션**: OS 취약점 스캐닝 활성화, 암호화 적용 등

## 2. 자동화 스크립트 및 알림 시스템

반복적인 관리 작업을 자동화하여 운영 부담을 줄이고 비용을 최적화합니다. **OCI Functions** (서버리스)를 사용하여 스크립트를 구현하는 것을 권장합니다.

- **비용 알림**:
    - **Budgets**: 예산의 80%, 100% 도달 시 담당자에게 이메일 또는 Slack/Teams로 알림을 자동 발송합니다.
- **리소스 태그 관리 자동화**:
    - **태그 미적용 리소스 탐지**: 태그 없이 생성된 리소스를 주기적으로 스캔하여, 생성자(Owner)에게 태그를 입력하도록 알림을 보냅니다.
    - **태그 강제화**: (심화) 리소스 생성 API 호출 시 태그가 없으면 생성을 거부하는 정책을 구현할 수 있습니다.
- **유휴 자원 관리 자동화**:
    - **자동 중지/종료**: 특정 시간(예: 저녁 9시) 이후 사용되지 않는 `dev`, `test` 컴파트먼트의 VM 인스턴스를 자동으로 중지(Stop) 또는 종료(Terminate)하는 스크립트를 실행합니다.
    - **장기 미사용 자원 정리**: 30일 이상 `Stopped` 상태인 VM이나 `Detached` 상태인 볼륨을 리포트로 만들어 담당자에게 정리 요청을 보냅니다.

## 3. 모니터링 및 리포팅

- **대시보드**: 비용, 사용량, 보안 상태를 한눈에 볼 수 있는 대시보드를 구성합니다.
    - 월별/주별/팀별 비용 지출 현황
    - 서비스별 사용량 변화 추이 (예: 컴퓨트 시간, 스토리지 용량)
    - 보안 경고 현황 (Cloud Guard 감지 항목)
- **정기 회의**: 매월 또는 분기별로 **비용/사용량/보안 상태 리뷰 미팅**을 운영하고, 결과를 모든 관련 팀과 공유합니다.
- **KPI 설정**: 운영 효율성을 측정하기 위한 핵심 성과 지표(KPI)를 설정하고 추적합니다.
    - `월별 테스트 컴파트먼트 평균 비용 <= 20만원`
    - `유휴 리소스 자동 종료율 >= 90%`
    - `태그 미적용 리소스 비율 <= 5%`

## 4. 워크로드별 정책

- **제품 테스트 환경**:
    - 모든 리소스는 '임시' 자원으로 간주합니다.
    - 테스트 완료 시 생성된 모든 리소스는 **반드시 종료**하는 것을 원칙으로 합니다.
    - 필요한 경우, 데이터를 보존하기 위해 종료 전 스냅샷 또는 아카이브 백업을 수행합니다.
- **운영 서비스 환경**:
    - 고가용성(HA), 재해복구(DR), 백업, 모니터링, 보안 정책 준수가 필수입니다.
    - Oracle DB 또는 Tibero와 같은 핵심 데이터베이스에 대한 백업 및 DR 전략을 반드시 문서화하고 정기적으로 테스트해야 합니다.

## 5. CDC 제품 테스트 환경 특화 고려사항

- **비용 누수 방지**: CDC 제품 테스트는 인스턴스를 생성하고 방치할 경우 비용이 빠르게 누적될 수 있습니다. 위에서 언급한 **유휴 자원 자동 종료/중지 스크립트**의 적용이 다른 어떤 환경보다 중요합니다.
- **DB 중심 워크로드 비용 최적화**:
    - OracleDB, Tibero 등은 컴퓨트 외에 스토리지, 백업, 네트워크 비용이 상당 부분을 차지합니다.
    - **스토리지 계층화**: 장기 보관 데이터는 비용이 저렴한 **아카이브 스토리지(Archive Storage)**로 이동시키는 정책을 고려합니다.
    - **백업 라이프사이클**: 백업 보존 기간을 데이터의 중요도에 따라 차등 적용하고, 오래된 백업은 자동으로 삭제되도록 설정합니다.
