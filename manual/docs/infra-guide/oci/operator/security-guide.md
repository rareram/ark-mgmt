# OCI 보안 가이드

이 문서는 OCI 환경의 보안을 강화하기 위한 네트워크, 데이터베이스 접근, 로깅 및 거버넌스 정책에 대한 핵심 가이드입니다.

## 1. 네트워크 및 DB 접근 보안

### VCN (Virtual Cloud Network) 설계

- **네트워크 분리**: 워크로드의 특성에 따라 VCN과 서브넷을 논리적으로 분리합니다.
    - **예시**: 운영(Production) 환경과 개발(Development) 환경의 VCN을 분리하여 상호 접근을 통제합니다.
    - **서브넷 분리**: VCN 내에서도 Public Subnet과 Private Subnet을 명확히 구분하여 사용합니다.
        - **Public Subnet**: 외부 인터넷과의 통신이 반드시 필요한 리소스(예: Public Load Balancer, Bastion Host)만 배치합니다.
        - **Private Subnet**: 데이터베이스, 애플리케이션 서버 등 내부 시스템은 모두 Private Subnet에 배치하여 외부 노출을 원천 차단합니다.

### 데이터베이스 접근 제어

- **Private Endpoint**: 모든 데이터베이스(Oracle DB, Tibero 등)는 **Private Endpoint**를 사용하도록 구성하여, VCN 내부에서만 접근 가능하도록 설정합니다.
- **Bastion Host**: 외부에서 Private Subnet의 데이터베이스나 서버에 접근해야 할 경우, Public Subnet에 위치한 **Bastion Host**를 통해서만 접근하도록 아키텍처를 구성합니다. 직접적인 DB 포트 노출은 금지합니다.

### 보안 그룹 (Security Groups)

- **최소 권한 원칙 적용**: 보안 목록(Security Lists) 및 네트워크 보안 그룹(NSG)을 사용하여, 애플리케이션 구동에 필요한 최소한의 포트와 프로토콜만 허용합니다.
- **Stateful vs. Stateless**: 트래픽의 특성을 이해하고 Stateful 규칙을 기본으로 사용하되, 고성능이 필요한 특정 시나리오에서는 Stateless 규칙을 검토합니다.

### 인증서 관리

- **TLS 적용**: 모든 외부 통신은 TLS(HTTPS)를 적용하여 암호화합니다.
- **인증서 관리**: 내부 인증 체계를 설계하고, 인증서의 정기적인 갱신 및 만료 알림 프로세스를 수립합니다. OCI Certificate Service 또는 HashiCorp Vault와 같은 도구 사용을 고려합니다.

## 2. 로그, 감사, 모니터링

### 중앙화된 로깅 및 감사

- **Audit 로그**: 모든 주요 활동(리소스 생성/변경/삭제, 사용자 로그인, 정책 변경 등)은 **OCI Audit** 로그를 통해 기록됩니다. 이 로그는 최소 1년 이상 보관하도록 설정합니다.
- **로그 집계**: 각 서비스 및 인스턴스에서 발생하는 로그(애플리케이션 로그, 시스템 로그 등)는 **OCI Logging** 서비스로 중앙화하여 집계하고 분석합니다.

### 이상 행위 탐지 및 알림

- **알림 설정**: **OCI Events** 및 **Notifications** 서비스를 사용하여 특정 이벤트 발생 시 보안 담당자에게 알림을 보냅니다.
    - **알림 예시**:
        - IAM 정책 변경 시도
        - 루트 사용자 로그인 시
        - 보안 그룹의 Any-to-Any(0.0.0.0/0) 규칙 추가 시
        - 특정 API 호출 실패가 반복될 때

### 정기적인 보안 점검

- **자동 점검**: **OCI Cloud Guard**를 활성화하여 OCI 모범 사례에 위배되는 보안 설정을 자동으로 탐지하고 경고를 받습니다.
- **수동 점검**: 분기별로 권한 오남용, 미사용 유저, 과도한 방화벽 규칙 등을 점검하고, 감사 보고서를 작성하여 관련 팀에 공유합니다.

## 3. 보안 거버넌스 및 정책

### 보안 체크리스트

- **리소스 생성 시**: 개발자가 리소스를 생성하기 전, 아래 항목들을 점검하도록 체크리스트를 요구합니다.
    - 데이터 암호화 여부 (Block Volume, Object Storage 등)
    - 백업 정책 적용 여부
    - 접근 제어 규칙의 적절성 (Public IP 사용 여부 등)
    - 필수 태그(`Owner`, `Project` 등) 적용 여부

### 데이터 민감도에 따른 보안 수준 차등 적용

- 테스트 환경의 리소스일지라도, 만약 민감한 개인정보나 고객 데이터를 포함하고 있다면 운영(Production) 환경과 동일한 수준의 보안 정책(암호화, 접근 제어 등)을 적용하도록 정책화합니다.

### 정기 리소스 감사

- 분기별 또는 반기별로 전체 리소스에 대한 감사를 실시하여, 소유자가 불분명하거나 더 이상 사용되지 않는 리소스를 식별하고 정리합니다.
