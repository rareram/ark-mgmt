# OCI 거버넌스 프레임워크

이 문서는 OCI 클라우드 환경을 효과적으로 관리하기 위한 구조, 비용, 보안, 운영에 대한 핵심 거버넌스 가이드입니다.

## 1. 구조 및 거버넌스 설계

### 테넌시·컴파트먼트 구조

첫 단계로 OCI의 테넌시(Tenancy) 안에서 컴파트먼트(Compartments) 를 논리적으로 설계해야 합니다. 이는 리소스 격리뿐 아니라 비용 배분, 책임 소재, 접근 제어 설계의 기반이 됩니다.

**기본 구획 예시:**

- `prod` (운영 서비스)
- `test` (제품 테스트)
- `dev` (개발/실험)
- `infra-common` (공통 인프라: 네트워크, 모니터링, 로그 등)

각 컴파트먼트 안에서도 제품별, 팀별, DB 종류별로 하위 구획을 구성하여 관리 단위를 더 세분화할 수 있습니다.

> **Oracle 권장 사항:** "Implement a Compartment Structure to Help Manage Costs"

### 태깅 (Tagging) 및 비용추적 구조

리소스 생성 시 반드시 **비용 추적용 태그(cost-tracking tag)**를 적용하도록 정책화해야 합니다. 태그가 없으면 어떤 팀이나 목적이 비용을 유발했는지 추적이 불가능합니다.

- **태그 네임스페이스:** `company`
- **태그 키:** `project`, `env`, `team`
- **태그 값:** `CDC-core`, `CDC-test`, `service-XYZ`, `prod`, `dev` 등

태그 기반으로 비용 분석, 리포팅, 내부 책임 배분 기준을 마련합니다.

### 정책·책임할당

- 각 컴파트먼트/프로젝트 별 **책임자(Owner)**를 지정합니다. (예: `CDC-test` 컴파트먼트 → QA팀 책임자)
- 리소스 생성, 비용 초과 알림, 접근 변경 등에 대한 책임 프로세스를 문서화합니다.
- 책임자는 매월 지출/사용 리포트를 검토하고, 식별된 낭비 리소스(Idle Resource)에 대한 **정리 및 종료를 주도**해야 합니다.

### 거버넌스 - 리소스 승인 및 라이프사이클

- **사전 승인 프로세스**: 중요한 리소스(DB, 고성능 컴퓨트 등)는 생성 전 담당자의 승인을 받도록 워크플로우를 도입합니다.
- **자동 종료 정책**: 사용이 끝난 테스트 인스턴스는 2주간 미사용 시 자동 종료 또는 아카이브되도록 라이프사이클을 정의합니다.

## 2. 비용 및 지출관리

### 예산 모니터링

- **OCI Budgets 기능**을 이용해 컴파트먼트 또는 태그 단위로 예산을 설정하고, 예산의 80%, 100% 도달 시 담당자에게 알림을 받도록 구성합니다.
- **강제 중단**: 예산 도달 시 리소스 생성을 강제로 막으려면 Quota 정책과 OCI Function을 결합한 별도 자동화 구현이 필요합니다. 모니터링과 알림을 기본으로 합니다.

### 비용 분석 및 인과관계 추적

- **OCI Cost Analysis** 기능을 활용해 서비스별/컴파트먼트별/태그별 비용 내역을 시각화하고 분석합니다.
- 매월 **"가장 비용이 많이 발생한 리소스 TOP 5"** 리포트를 작성하고 다음을 점검합니다.
    - 사용량 대비 오버프로비저닝 여부
    - 사용하지 않는(Idle) 인스턴스 또는 스토리지 존재 유무
    - 종료되지 않은 테스트 리소스가 계속 비용을 발생시키는지 여부

### 자동화된 리소스 종료 및 권고

- 테스트 환경 인스턴스는 **밤 11시 이후 자동으로 중지**되도록 스케줄링합니다.
- 장시간 `Stopped` 상태인 인스턴스나 `Detached` 상태인 블록 볼륨을 주기적으로 탐지하여 정리합니다.

### 비용 책임 배분

- 태그를 기준으로 "누가 얼마를 썼는가"를 투명하게 하여 내부 회계/청구 항목으로 연결합니다.
- 월 단위로 각 프로젝트 책임자에게 비용 리포트를 보내고, 필요 시 비용 사용 계획을 요청합니다.

## 3. 접근, 권한, 보안

### IAM 설계

**최소 권한(Least Privilege) 원칙**에 따라 역할(Role)별로 IAM 정책을 설계합니다.

- **운영팀**: `prod` 컴파트먼트에 대한 읽기/쓰기 권한
- **개발/QA팀**: `test`/`dev` 컴파트먼트에 대한 권한
- **공통 인프라팀**: `infra-common` 컴파트먼트 권한

리소스 생성·삭제와 같은 중요한 권한은 책임자 승인 하에 부여하도록 정책화합니다.

### 네트워크 및 DB 접근 보안

- **VCN 설계**: 테스트용과 운영용 네트워크를 논리적/물리적으로 분리합니다.
- **DB 접근**: DB는 Public IP 노출을 피하고, 사내 VPN 혹은 **Bastion 호스트**를 통해서만 접근하도록 구성합니다.
- **보안 그룹 (NSG, Security List)**: 허용된 IP와 포트만 최소한으로 개방합니다.

### 로그, 감사, 모니터링

- **Audit 로그**: 모든 주요 액션(리소스 생성/변경/삭제) 로그를 중앙 저장소(OCI Logging)로 집계하고 1년 이상 보관합니다.
- **이상 행위 탐지**: 비정상적인 접근 시도, 권한 없는 작업 실패 등의 로그를 감지하여 보안팀에 알림을 보냅니다.
- **정기 보안 점검**: 분기별로 취약점, 권한 오남용 등을 점검하고 결과를 문서화합니다.

## 4. 운영 및 자동화

### IaC (Infrastructure as Code) 및 Landing Zone

- 수작업 오류를 줄이고 표준화된 환경을 제공하기 위해, 초기 OCI 환경(네트워크, IAM, 보안 정책 등)을 **Terraform** 코드로 관리되는 **Landing Zone**으로 구성합니다.

### 자동화 스크립트 및 알림 시스템

- **태그 미적용 리소스 자동 식별**: 태그가 없는 리소스가 생성되면 담당자에게 알림을 보내 태그 입력을 강제합니다.
- **유휴 자원 자동 종료**: 특정 시간 이후 사용되지 않는 테스트 VM은 OCI Function을 통해 자동 중지/종료시킵니다.

### 모니터링 및 리포팅

- **정기 회의**: 매월 비용/사용량/보안 상태 리뷰 미팅을 운영하고 결과를 공유합니다.
- **KPI 설정**: "월별 테스트 컴파트먼트 평균 비용 ≤ N원", "유휴 리소스 종료율 ≥ 90%" 등 측정 가능한 목표를 설정합니다.

## 6. 제품 테스트 환경 특화 고려사항 (CDC 제품 관점)

- **비용 최적화**: 테스트 인스턴스 생성 후 방치되지 않도록 자동화 스크립트(유휴 자원 탐지 및 종료)가 특히 중요합니다.
- **DB 중심 워크로드**: OracleDB, Tibero 등은 스토리지, 백업 비용이 크므로, 데이터 등급에 따른 스토리지 계층(Archive Storage 등) 활용 및 백업 자동 만료 정책을 수립해야 합니다.
- **리소스 생성 템플릿**: 팀의 자율성은 보장하되, 태깅/보안 옵션 등이 미리 정의된 '리소스 생성 템플릿'을 사용하도록 유도하여 거버넌스를 준수하게 합니다.
