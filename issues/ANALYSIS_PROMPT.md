# 🤖 지라 이슈 분석 자동화 프롬프트

이 문서는 `issues/` 폴더에 수집된 JSON 데이터를 분석하고 보고서를 생성한 뒤, 데이터를 정리하는 전체 워크플로우를 정의합니다.

---

## 📋 실행 방법 (Gemini CLI에 복사/붙여넣기)

> "issues/ANALYSIS_PROMPT.md에 정의된 '지라 이슈 분석 워크플로우'를 실행해줘"

---

## 🛠️ 지라 이슈 분석 워크플로우 정의

### 1. 대상 데이터 수집
- `issues/*.json` 패턴의 모든 파일을 분석 대상으로 삼는다.

### 2. 분석 참조 및 철학 (Context)
- **소통의 중심점**: SE의 '현상 언어'와 개발팀의 '코드 언어'를 연결하고, 이사님이 임원/영업에게 보고할 수 있도록 중간 언어로 번역한다.
- **정직한 분석 (No Fluff)**: 억지로 추론하여 그럴듯하게 꾸며내지 않는다. 정보가 부족하면 **"정보 부족으로 알 수 없음"**을 명확히 명시한다.
- **재현성 중시**: 증상 재현이 가능한지 여부를 분석의 핵심 기준으로 삼는다.
- **제품 컨텍스트**: AFC(Redo 역설계), AFO(물리복제), AFF(파일복제)의 특성을 고려한다. 특히 보안이 철저한 고객사(은행, 관공서 등) 환경임을 감안하여 한정된 정보에서 논리적 가설을 세우되, 가설임을 밝힌다.

### 3. 작업 프로세스
1. **내용 파악**: JSON의 `summary`, `description`, `comments`를 대조하여 SE가 본 현상과 개발팀의 의견 차이를 분석한다.
2. **원인 분류**: `impact-categories.md` 기준에 따라 [솔루션/고객환경/작업자행위/일시적현상/알수없음] 중 하나로 분류한다.
3. **분석서 작성**: 
   - 경로: `issues/analysis/{{YYYY-MM-DD}}-{{ISSUE_KEY}}-analysis.md`
   - **분석 신뢰도** 점수를 매기고, 추가로 필요한 데이터가 무엇인지 제안한다.
4. **데이터 정리**:
   - 분석이 완료된 JSON 파일은 `issues/jira-analysis/` 폴더로 이동(Move)시킨다.

### 4. 보고서 작성 가이드 (Next Action 중심)
- **개발팀**: **[수정 필요성]** 코드 레벨의 수정이 필요한지, 아니면 우회 방법(Workaround)을 제시할 것인지 명확히 한다.
- **임원**: **[리스크 및 자원 투입]** 제품 결함일 경우 패치 일정과 리소스 투입 계획을, 환경 문제일 경우 고객사와의 협상 및 책임 소재를 명확히 보고한다.
- **영업/SE**: **[고객 대응 논리]** 
  - **솔루션 문제 시**: "업그레이드/패치로 해결 예정"임을 알리고 신뢰를 유지한다.
  - **고객 환경 문제 시**: "우리 제품은 정상이나 고객사 DB/네트워크 설정 변경이 필수적"임을 논리적으로 설명하여 고객을 설득할 근거를 제공한다.
  - **작업자 실수 시**: 가이드라인 재교육 및 방어 설정 추가 계획을 안내한다.
